## Wake-on-motion detection and data logging application

**[Sasquatch](https://www.tindie.com/products/tleracorp/sasquatch-stm32wb5mmg-development-board/)** is an ultra-low-power, BLE-enabled development board with an [STM32WB5MMG](https://www.st.com/en/microcontrollers-microprocessors/stm32wb5mmg.html) module (STM32WB55) MCU host, an ST LIS2DW12 3-axis accelerometer and a 16-MByte Macronix MX25L12835FZNI QSPI flash memory. It is the minimal setup for practical wake-on-motion detection and data logging applications. These include monitoring a door for entry/exit events, monitoring infrequently-moved equipment at a job site, earthquake detection and, more commonly, package state monitoring during shipment.

In the latter case, purchasers of high-value products (and their insurance companies) want to know if their package in transit has 
seen extremes of temperature, has been dropped or tipped over during handling, or generally suffered an event resulting in damage. The data desired includes a record of the package temperature, the inertial forces acting on the package during the event, and a time/date stamp. Sasquatch can provide these data (and other data if additional sensors are added), and this is a minimally useful data set to offer a practical, battery-powered application that can cover a year or longer monitoring time.

The heart of such an application is the [LIS2DW12](https://www.st.com/en/mems-and-sensors/lis2dw12.html) 3-axis accelerometer. In the Sasquatch device, the wake-on-motion and sleep-on-no-motion interrupts are set up to wake the MCU from one of two ultra-low-power states, either STOP or STANDBY. In STOP, the RTC and all SRAM remains active (state-preserving) and the device rapidly goes back to the RUN state on any interrupt, including the RTC Alarm interrupt. In STANDBY, the RTC and only 32 kBytes of SRAM remain active (partially state preserving) and the device takes a bit longer to return to the RUN state. Also, the MCU can only wake from interrupt in STANDBY from a few special "wakeup" GPIOs, like the one (PC2 = Arduino D33) we chose for the LIS2DW12 interrupt. The details can be found in the STM32WB55 data sheet. In practice, STANDBY uses ~1.5 uA less power than STOP, so for applications with very low duty cycles like a motion-detect data logger, STANDBY should result in lower average current usage.

If we only wanted to wake the MCU on any motion and nothing else, we would run the LIS2DW12 at the lowest sample rate (1.6 Hz in stationary mode) and lowest power mode (power mode 1 with 12-bit data). The sensor runs continuously in this configuration at ~0.4 uA sending interrupts to wake the MCU on any motion above a programmable threshold and to allow the MCU to go back to sleep on detection of no motion for a programmable time period. However, for a motion detect data logger application, we are interested in more than this.

The LIS2DW12 has an embedded 32-sample (3-axes of acceleration x 32 = 92 int16_t words of raw data = 192 bytes in total) FIFO (first in, first out) memory that can store data continuously at the programmed sample rate where the watermark fill level (0 - 31) is programmable and can be triggered by interrupt. That is, when a wake-on-motion interrupt occurs, the FIFO operation mode is changed from continuous to "FIFO" (CONT-TO-FIFO) mode where data is collected until the FIFO is full (watermark is reached, *i.e.*, 32 samples). This allows the state of motion just at the wake event and after to be captured for data logging onto the QSPI flash memory along with a time/date stamp, the battery voltage, and the MCU (and/or LIS2DW12) temperature.

*In this test, the Sasquatch (mounted in a breadboard) was picked up by hand and placed back on a flat table all in a few seconds. The FIFO captures this motion from the first data point after at least one of the accelerometer values exceeded 62.5 mg and lasting for 32/12.5 Hz = 2.56 seconds.*

We are taking advantage of the embedded RTC to produce a time stamp for logging to go along with the accel data, temperature and battery voltage. The RTC also has an alarm which we are using to provide a "heartbeat" for the device. The alarm can be set to wake every second, every minute, every hour, once per day, *etc*. Once per hour might be the best choice for this application. 

The 16 MByte flash memory is huge;  we are writing 192 bytes of acceleration data and 10 bytes of time/data, temperature and battery voltage data per wake event, so "wasting" 54 bytes of capacity per 256-byte page. We can record 65,535 wake events (pages) of 202-byte data. Even if we needed to record one event per minute (an absurdly high duty cycle for this application), we wouldn't fill up the flash for 45 days. At a more reasonable and likely one motion event per hour, the flash capacity would last more than seven years! Since the temperature sensors embedded in the MCU and the LIS2DW12 have no capability of threshold setting or interrupting the MCU, logging the temperature and time/date with the RTC alarm heartbeat in a section near the end of the flash memory would be a useful way to keep track of temperature excursions not accompanied by significant motion.

In these sketches, we have chosen the full-scale range to be 2 g, the sample rate to be 12.5 Hz, and lowest power mode (power mode 1), which allows ~2.6 seconds of 12-bit accel data to be collected for each wake event whenever the threshold exceeds 62.5 mg. The resultant ~1 mg resolution, narrow range, and slow data rate is good enough to capture human-scale motions, but can be adjusted for more demanding motion capture at the cost of extra power usage. The time delay before sleep is allowed to resume is as small as possible at this data rate, 1.28 sec. The reason for using both a wake-on-motion and a sleep-on-no-motion interrupt is so the FIFO behavior can be controlled; it has to be cleared (put in BYPASS mode) after the FIFO is read upon an LIS2DW12 wake event and the MCU is logging data. Then after the no motion activity time out, the FIFO has to be reset to CONT-TO-FIFO mode to allow the next wake event to be captured. We also disable the wake-on-motion interrupt once a wake event is triggered and then re-enable once the sleep-on-no-motion interrupt is triggered to avoid multiple wake-on-motion interrupts for the same wake event. All this fussing allows each motion event to be captured cleanly, minimizes the time the MCU is awake and logging data, and results in the lowest average power usage we can achieve for this device.

In testing with the LIS2DW12 running continuously at 12.5 Hz, we measured total current usage of 6.85 uA in STOP mode and 4.25 uA in STANDBY mode. The difference is a bit larger than the ~1.5 uA we measured when testing the MCU by itself, possibly due to the attached interrupt in STOP mode that is absent in STANDBY (which uses a different mechanism for managing the wakeup pin). For a motion event detection and data logging test, I ran the above programs for ten minutes with the RTC alarm heartbeat every minute and moving the breadboard-mounted Sasquatch five different times during the test. The average current usage over the ten minutes was somewhat surprisingly 8.5 uA for the sketch using STOP and 9.2 uA for the sketch using STANDBY. This despite the 2.6 uA lower STANDBY current usage than STOP in the no-motion case! This is likely due to the fact that in wake from STANDBY, all of the variables and arrays, I2C, etc have to instantiated while these persist in waking from STOP. So there is a duty cycle crossover above which using STOP mode wins and below which using STANDBY wins because of this extra overhead.

This is easy to estimate. The average current usage using STOP mode is 8.5 - 6.85 = 1.65 uA for five motion events, so 0.33 uA per motion event and the average current usage using STANDBY is 9.2 - 4.25 = 4.95 uA for five motion events, so 0.99 uA per motion event. For what N does 6.85 + N * 0.33 equal 4.25 + N * 0.99? Turns out to be N = 4 motion events every ten minutes. So if the number of motion events that need to be detected on average is less than 24 per hour (highy likely), using STANDBY will offer the lowest average power usage. This all remains to be tested over longer time scales, of course, but it is clear that this minimal motion event detection and data logging application can be implemented simply at an average power usage << 10 uA.

Running this application on Sasquatch for a year would require a battery capacity < 90 mAH, i.e., a cheap [CR2016 coin cell](https://www.batteryjunction.com/products/panasonic-cr2016-lithium-coin-cell-battery-bulk?variant=47570227724520&msclkid=939620cadc741dc8104c49305e2a9da0).
